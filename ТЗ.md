Техническое Задание: Feedback-Loop Service

Стек: Python (aiogram 3.x), PostgreSQL (SQLAlchemy 2.0 Async), Redis, OpenAI Whisper API, Google Sheets API.
1. Общее описание системы

Сервис предназначен для централизованного сбора учебных/творческих работ и организации процесса их проверки. Система поддерживает три режима: классическая экспертная проверка, анонимное Peer-to-Peer (P2P) рецензирование и конкурсное голосование. Все взаимодействие происходит через Telegram-бота.
2. Ролевая модель

    Организатор: * Создает и настраивает "Сборы" (кампании).

        Управляет доступом (бан/разбан).

        Настраивает критерии и TTL проверок.

        Выгружает результаты в Google Sheets.

    Студент (Автор/Рецензент):

        Регистрируется в системе (ФИО, Группа).

        Загружает свои работы.

        В режиме P2P выступает в роли проверяющего для других.

    Эксперт:

        Имеет доступ к очереди работ.

        Оставляет обратную связь (текст/голос/видео-кружок).

3. Архитектура базы данных (SQLAlchemy)
3.1. Сущности

    User: tg_id (PK), full_name, study_group, role (Enum), is_banned (Bool).

    Campaign (Сбор): id (PK), title, type (Expert/P2P/Voting), min_score, max_score, ttl_minutes, is_expert_anon (Bool), is_active (Bool).

    Submission (Работа): id (PK), campaign_id (FK), author_id (FK), file_id (TG file_id), status (Enum: uploaded, in_review, reviewed, rejected).

    Review (Рецензия): id (PK), submission_id (FK), reviewer_id (FK), score (Int), comment_text (Text), voice_file_id (TG file_id, Optional).

4. Функциональные модули
4.1. Модуль управления очередью (Redis + PostgreSQL)

Для исключения дублирования проверок реализуется механизм блокировок:

    Запрос работы: Эксперт запрашивает работу -> SQL выбирает Submission со статусом uploaded.

    Блокировка: В Redis записывается ключ active_review:{submission_id} со значением reviewer_id и временем жизни TTL.

    Статус: В БД статус меняется на in_review.

    Освобождение: * При завершении проверки: ключ удаляется, статус в БД -> reviewed.

        При нажатии "Вернуть в очередь": ключ удаляется, статус в БД -> uploaded.

        При истечении TTL: фоновая задача (или проверка при следующем запросе) возвращает статус uploaded.

4.2. Модуль Peer-to-Peer (P2P)

    Условие участия: Кнопка "Проверить других" доступна только после успешной загрузки собственной работы (status != None).

    Анонимность: При выдаче работы рецензенту скрывается ФИО автора.

    Исключение селф-проверки: SQL-запрос при выдаче: WHERE author_id != current_user_id.

4.3. Модуль транскрибации (Whisper API)

    Вход: Voice-сообщение или Video Note.

    Процесс:

        Скачивание файла на сервер.

        Если длительность превышает лимит API (25 МБ), нарезка файла на части.

        Отправка в OpenAI Whisper API.

        Объединение текста и сохранение в Review.comment_text.

    Выход: Текст комментария дублируется в Google Таблицу.

4.4. Интеграция с Google Sheets

    При переходе работы в статус reviewed, данные асинхронно отправляются в таблицу.

    Поля: [ID работы, Время, Кампания, Автор, Группа, Проверяющий, Балл, Текст комментария].

    Если эксперт редактирует оценку через бота, соответствующая строка в таблице обновляется.

5. Алгоритм взаимодействия (User Flow)
Сценарий студента:

    /start -> Ввод ФИО и Группы (сохранение в User).

    Выбор активного сбора -> Загрузка файла (сохранение file_id в Submission).

    (Если P2P) -> Кнопка "Проверить 3 работы" -> Получение анонимных ссылок/файлов.

Сценарий эксперта:

    Кнопка "Взять работу" -> Бот выдает файл и кнопки с баллами (от min до max).

    Оценка -> Отправка комментария (текстом или голосом).

    Подтверждение -> Данные улетают в БД и Google Sheets, статус работы закрывается.

6. Нефункциональные требования

    Интерфейс: Исключительно Inline-кнопки (минимизация ручного ввода).

    Безопасность: Проверка на is_banned при каждом действии. Валидация типов файлов (например, только .pdf или .docx, если задано организатором).

    Логирование: Хранение логов критических ошибок (ошибки API Whisper, ошибки записи в Sheets) в отдельный файл или таблицу.

    Хранение: Файлы не хранятся локально дольше времени, необходимого для обработки Whisper (экономия места на диске).

7. Обработка исключений (Edge Cases)

    Эксперт "пропал": Если работа висит в in_review дольше TTL, она автоматически возвращается в пул.

    Битый файл: Эксперт нажимает "Rejected" -> Студент получает уведомление: "Ваша работа отклонена, загрузите файл заново".

    Лимиты Telegram: Использование aiogram очереди для рассылки уведомлений студентам о результатах проверки, чтобы не поймать Flood Limit.

План реализации (Priority):

    MVP 1: Регистрация + Загрузка файла + Экспертная проверка (Балл + Текст) + Google Sheets.

    MVP 2: Механика очередей в Redis (TTL и возврат в пул) + Статус "Rejected".

    MVP 3: Интеграция Whisper (Голос -> Текст) + Обработка "кружочков".

    MVP 4: Режим P2P с проверкой условий сдачи.

    Final: Панель организатора (Бан-лист, создание кампаний через бота).
